## 四、开发前技术决策清单

以下决策应在 T1.1 完成，记录在 `docs/DECISIONS.md`：

| # | 决策项 | 选项 | 推荐 |
|---|--------|------|------|
| 1 | 主语言 | Go vs Python | **Go**（单文件分发，无运行时依赖） |
| 2 | Go Web框架 | gin / fiber / chi / net/http | **fiber**（性能好，API风格简洁） |
| 3 | 前端方案 | Vue.js SPA / Alpine.js+HTMX | **Alpine.js+HTMX**（一期轻量，二期可换Vue） |
| 4 | 日志库 | slog / zap / zerolog | **slog**（Go 1.21+标准库，零依赖） |
| 5 | 配置格式 | YAML / TOML / JSON | **YAML**（用户可读性最佳） |
| 6 | 认证方案 | Session Cookie / JWT | **JWT**（无状态，适合单机部署） |
| 7 | frp集成方式 | 子进程 / CGO嵌入 / 纯Go重写 | **子进程**（最稳定，隔离性好） |
| 8 | MC交互协议 | RCON / stdin管道 / mcrcon | **RCON**（标准协议，功能完整） |
| 9 | 打包工具 | Go: goreleaser / Python: PyInstaller | 取决于主语言选择 |
| 10 | 最低Java版本 | Java 17 vs Java 21 | **Java 17**（兼容MC 1.18+） |

- **分支策略**：`main` 保持可发布状态；每个阶段在 `feature/stage-N` 分支开发，完成里程碑后合并。
- **提交规范**：`[T1.3] feat(config): 支持YAML配置热重载`
- **测试要求**：核心模块（进程管理、代理层、认证）单元测试覆盖率 ≥ 70%。
- **日志格式**：结构化日志，每条必含 `ts`（时间戳）、`level`、`msg`、`module` 字段。
